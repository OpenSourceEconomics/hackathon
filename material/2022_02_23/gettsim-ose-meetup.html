About GETTSIM

  • Depiction of the German Taxes and Transfers System
  • Open source, collaborative effort, IZA is main driver behind it
  • Inputs: Household composition, gross incomes of all members, wealth, consumption, …
  • Outputs: Taxes paid, benefits and transfers received

Introductory Example

Two use cases: 1) Policy advice/ micro simulation

  • Ex-ante evaluation of potential reforms
  • Often short-term projects
  • No behavioural model (morning-after evaluation) or very simple one

  • Key challenges of this project

Challenge 1: Heterogeneous user base

  • Hide technical functionality from user:
    • Small functions which can be read by non-Python experts
    • policy parameters specified in .yaml-files (Example)
  • Tutorials
  • Code documentation
  • (Interface to other languages)

Challenge 2: Model policy changes

  • Taxes and transfers system is constantly evolving in many dimensions

  • Users introduce their own (potential) reforms

  • -> Need for flexibility

  • Solution: Directed Acyclic Graph (DAG)

Challenge 2: Model policy changes

  • Each function:

    • Calculates exactly one variable/column

    • Set of arguments, all one of the following:

      • User-provided input variables
      • Outputs of other functions
      • (policy parameters)

Challenge 2: Model policy changes

  • User specifies targets she wants to be computed
  • GETTSIM builds DAG containing all functions necessary to calculate these targets
  • GETTSIM executes functions in the correct order
  • Very easy to:
    • replace a function
    • replace a set of functions
    • add new functions

Challenge 3: Smooth functions

  • Taxes and transfers are typically rounded (e.g. income taxes rounded down to full euro)
  • Some applications require the rounded amount as specified in the law. This is also helpful for creating test cases.
  • Other applications benefit if functions are mostly smooth (numerical optimisation).
  • -> make rounding optional

Challenge 3: Smooth functions

  • Specify rounding parameters in parameter .yaml files
rounding:
  eink_st_tu:
    1984-01-01:
      base: 1
      direction: down
      reference: § 32a Abs. 1 S. 6 EStG

Challenge 3: Smooth functions

Challenge 4: Speed

  • Current implementation: work on pandas Series

    out = pd.Series(0, index=schwerbe_ausweis_g.index)
out.loc[schwerbe_ausweis_g] = mehrbedarf_singles

  • Problems:

    • Code is not very intuitive (especially for non-programmers)
    • Code is slow

Challenge 4: Speed

  • Solution:

    • Specify functions on scalars
    if schwerbe_ausweis_g:
  out = mehrbedarf_singles
else:
  out = 0
    • Vectorize using
      • numba (usually)
      • numpy (for debugging)
      • jax (even faster, but more difficult to set up)

Challenge 4: Speed

  • Especially costly: transformation between household-level and individual-level variables
kindergeld_m_hh = kindergeld_m.groupby(hh_id).sum()
hh_id.replace(alter_jüngstes_kind)

  • Proposal:

    • hh-level variables defined on individual level
      • hence, no transformation “hh-level -> individual level” necessary
    • provide fast (automated) functions for aggregation on tu-level and hh-level
      • making use of numba/jax

Conclusion

  • Good coding practices help us to improve our code:
    • more intuitive to understand
    • less error-prone
    • more flexible
    • better suitable for high-performance computing applications
